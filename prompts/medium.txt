You are a senior distributed systems architect. A startup is building a real-time collaborative document editor (similar to Google Docs) that must support 10 million concurrent users across 5 geographic regions. The system must guarantee:

1. Sub-100ms latency for local edits
2. Eventual consistency with a convergence window under 2 seconds globally
3. Offline editing support with automatic conflict resolution upon reconnection
4. End-to-end encryption where the server never sees plaintext content
5. A plugin architecture allowing third-party developers to extend the editor

The current prototype uses a single PostgreSQL database and WebSocket connections, which obviously won't scale.

Please provide a comprehensive technical architecture covering the following aspects:

**Data Layer**: What CRDT variant would you use for the document model? Compare operation-based vs state-based CRDTs for this use case. How would you handle rich text formatting, embedded media, and concurrent table editing? Discuss the trade-offs between Yjs, Automerge, and Diamond Types.

**Networking**: Design the real-time sync protocol. How do you handle presence awareness (cursors, selections)? What transport layer would you use — WebSockets, WebTransport, or SSE? How do you implement the offline queue and reconnection logic? Describe your approach to delta compression for bandwidth optimization.

**Storage & Persistence**: How do you persist CRDT state efficiently? What's your strategy for document snapshots vs operation logs? How do you handle garbage collection of tombstones? Design the backup and point-in-time recovery system.

**Security**: How do you implement end-to-end encryption while still supporting real-time collaboration? Discuss key management, key rotation, and how you handle users joining/leaving a shared document. What encryption scheme allows the server to merge encrypted operations without decryption?

**Infrastructure**: Design the multi-region deployment. How do you route users to the nearest edge node? What happens during a region failover? How do you handle the "split brain" scenario? Describe your approach to capacity planning and auto-scaling.

**Plugin System**: How do you sandbox third-party plugins to prevent security issues while giving them useful access to the document model? Discuss your API surface, versioning strategy, and how you handle plugins that need to modify the CRDT state.

Be specific about technology choices, justify each decision, and discuss failure modes for each component.
