You are a senior distributed systems architect. A startup is building a real-time collaborative document editor (similar to Google Docs) that must support 10 million concurrent users across 5 geographic regions. The system must guarantee:

1. Sub-100ms latency for local edits
2. Eventual consistency with a convergence window under 2 seconds globally
3. Offline editing support with automatic conflict resolution upon reconnection
4. End-to-end encryption where the server never sees plaintext content
5. A plugin architecture allowing third-party developers to extend the editor

The current prototype uses a single PostgreSQL database and WebSocket connections, which obviously won't scale.

Please provide a comprehensive technical architecture covering the following aspects:

**Data Layer**: What CRDT variant would you use for the document model? Compare operation-based vs state-based CRDTs for this use case. How would you handle rich text formatting, embedded media, and concurrent table editing? Discuss the trade-offs between Yjs, Automerge, and Diamond Types.

For the CRDT implementation, consider the following constraints:
- Documents can contain up to 500 pages with mixed content types (text, tables, images, code blocks, mathematical equations)
- Tables can have up to 10,000 rows and 100 columns with formulas that reference other cells
- Users frequently copy-paste large sections (10,000+ characters) between documents
- Some documents have 50+ simultaneous editors in the same section
- The undo/redo history must be per-user and must work correctly even with concurrent edits from others
- Documents must support comments and suggestions (like Google Docs "suggesting" mode) that are themselves collaborative

**Networking**: Design the real-time sync protocol. How do you handle presence awareness (cursors, selections)? What transport layer would you use — WebSockets, WebTransport, or SSE? How do you implement the offline queue and reconnection logic? Describe your approach to delta compression for bandwidth optimization.

Consider these networking challenges:
- Users on mobile networks with 500ms+ latency and frequent disconnections
- Corporate firewalls that block WebSocket upgrades and require HTTP/2 fallback
- A user who was offline for 2 weeks reconnects with 50,000 pending operations while 200 other users have been actively editing
- Presence information must update within 200ms so cursor positions feel real-time
- The system must handle a "thundering herd" scenario where 100,000 users open the same document simultaneously (e.g., a company-wide announcement)

**Storage & Persistence**: How do you persist CRDT state efficiently? What's your strategy for document snapshots vs operation logs? How do you handle garbage collection of tombstones? Design the backup and point-in-time recovery system.

Address these storage requirements:
- Documents must be recoverable to any point in time within the last 365 days
- The operation log for a heavily-edited document can reach 10 million operations
- Snapshots must be taken without blocking active editing sessions
- Cold storage costs must be minimized — not all documents are equally active
- GDPR compliance requires the ability to permanently delete all content from a specific user across all document versions and operation logs within 30 days of request
- Search must work across all documents a user has access to, with results in under 500ms

**Security**: How do you implement end-to-end encryption while still supporting real-time collaboration? Discuss key management, key rotation, and how you handle users joining/leaving a shared document. What encryption scheme allows the server to merge encrypted operations without decryption?

Security constraints:
- Some customers require that encryption keys never leave their on-premise HSMs
- Key rotation must happen every 90 days without disrupting active sessions
- When a user is removed from a document, they must lose the ability to decrypt any future content within 60 seconds
- The system must pass SOC 2 Type II and ISO 27001 audits
- Admin users need the ability to perform content moderation without breaking E2E encryption (legal compliance for certain enterprise customers)
- Audit logs must capture who accessed what and when, without revealing document content

**Infrastructure**: Design the multi-region deployment. How do you route users to the nearest edge node? What happens during a region failover? How do you handle the "split brain" scenario? Describe your approach to capacity planning and auto-scaling.

Infrastructure constraints:
- The five regions are US-East, US-West, EU-West, AP-Southeast, and AP-Northeast
- Each region must be able to operate independently for up to 4 hours if inter-region connectivity is lost
- P99 latency for API calls must be under 200ms
- The system must handle a 10x traffic spike within 5 minutes (auto-scaling)
- Cost optimization: the system should scale down to minimal infrastructure during off-peak hours (weekends, holidays)
- Deployment must support both Kubernetes and bare-metal for customers who cannot use cloud providers

**Plugin System**: How do you sandbox third-party plugins to prevent security issues while giving them useful access to the document model? Discuss your API surface, versioning strategy, and how you handle plugins that need to modify the CRDT state.

Plugin system requirements:
- Plugins must not be able to access data from documents they are not installed on
- A misbehaving plugin must not affect editor performance (CPU, memory, network limits)
- Plugins need access to a subset of the CRDT API to read and modify document content
- The plugin marketplace must support code review and automated security scanning
- Plugins must work offline and sync their state when reconnecting
- Plugin developers need a local development environment with hot-reload
- The system must support both client-side plugins (UI extensions) and server-side plugins (webhooks, integrations)
- Backward compatibility: removing a plugin must not corrupt the document, even if the plugin added custom content types

For each section, provide specific technology choices with version numbers where relevant, justify each decision against alternatives you considered, describe at least two failure modes and their mitigations, and estimate the engineering effort required to build each component.
