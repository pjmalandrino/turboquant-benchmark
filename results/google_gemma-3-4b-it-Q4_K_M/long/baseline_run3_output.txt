## Real-Time Collaborative Document Editor Architecture

Here's a comprehensive architecture for the real-time collaborative document editor, addressing the specified requirements.

**1. Data Layer: CRDTs & Document Model**

* **CRDT Variant:** Operation-based CRDTs (specifically, Conflict-Free Replicated Data Types - CRDTs).
* **Operation-Based vs. State-Based:** Operation-based CRDTs are better suited here due to the complexity of the document model (rich text, tables, embedded media). State-based CRDTs would require complex state synchronization, which is difficult to manage with a large number of concurrent users and diverse content types.
* **Yjs:** Chosen as the core CRDT implementation. Yjs is a highly performant, battle-tested, and actively maintained CRDT library. It’s designed for collaborative editing and offers excellent support for rich text and complex data structures.
* **Automerge & Diamond Types:** While Automerge is a powerful CRDT library, Yjs offers better integration with web technologies and a more mature ecosystem. Diamond Types are useful for certain operations (e.g., merging), but Yjs handles the core CRDT logic more efficiently.
* **Document Model:** Yjs will represent the document as a graph of operations. Each operation represents a change to the document.
* **Rich Text Formatting:** Yjs natively supports rich text formatting through its Text CRDT.
* **Embedded Media:** Media will be represented as URLs within the document graph, with metadata (size, type) stored alongside the operation.
* **Concurrent Table Editing:** Yjs's Table CRDT handles concurrent table editing efficiently.
* **Undo/Redo:** Yjs's built-in undo/redo mechanism is efficient and handles concurrent edits correctly.
* **Operation Log:** A separate, optimized operation log (e.g., using a columnar database like ClickHouse) will be used for long-term storage and search.
